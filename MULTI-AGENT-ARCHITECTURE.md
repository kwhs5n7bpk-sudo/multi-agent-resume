# OpenCode 多 Agent 系统架构方案

> 设计者：老k | 版本：v1.0 | 日期：2026-07-07

---

## 一、系统总览

### 1.1 设计目标

基于 OpenCode 构建一个**角色化多 Agent 协作系统**，通过微信 Bot 作为用户入口，实现从需求到交付的全链路 AI 协作开发。

### 1.2 架构模式：Hub-Spoke（中心辐射）

```
┌──────────────────────────────────────────────────────────────────────┐
│                        微 信 用 户 端                                │
└────────────────────────────┬─────────────────────────────────────────┘
                             │ 微信消息
                             ▼
┌──────────────────────────────────────────────────────────────────────┐
│                    微信 Bot 桥接服务 (Node.js)                       │
│   ┌─────────────┐  ┌──────────────┐  ┌──────────────────────────┐  │
│   │ 消息收发     │  │ 会话管理      │  │ SSE 事件监听             │  │
│   │ (wechaty)   │  │ (Redis)      │  │ (长任务实时推送)          │  │
│   └──────┬──────┘  └──────┬───────┘  └────────────┬─────────────┘  │
│          └────────────────┼───────────────────────┘                │
│                           │ HTTP API                                │
└───────────────────────────┼────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────────────┐
│                   OpenCode Server  (opencode serve)                  │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │                    老k  (Primary Agent / 中枢调度)             │  │
│  │                                                               │  │
│  │   职责：意图理解 → 任务拆解 → 角色调度 → 结果汇总 → 质量把关   │  │
│  └───────┬─────────┬──────────┬──────────┬──────────┬────────────┘  │
│          │         │          │          │          │                │
│     Task │    Task │     Task │     Task │     Task │                │
│          ▼         ▼          ▼          ▼          ▼                │
│   ┌─────────┐┌─────────┐┌─────────┐┌─────────┐┌─────────┐          │
│   │ 产品PM  ││  UI设计  ││  前端   ││  后端   ││  测试   │          │
│   │subagent ││subagent ││subagent ││subagent ││subagent │          │
│   └─────────┘└─────────┘└─────────┘└─────────┘└─────────┘          │
│                                                              ┌─────────┐
│                                                              │  运维   │
│                                                              │subagent │
│                                                              └─────────┘
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│  子agent只能被老k通过Task工具调用，彼此不可直接通讯（严格hub-spoke）  │
└──────────────────────────────────────────────────────────────────────┘
```

### 1.3 核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **单一入口** | 所有外部请求经微信Bot → 老k，老k是唯一 Primary Agent |
| 2 | **严格层级** | 子agent之间禁止直接通讯，所有协作经老k中转 |
| 3 | **能力隔离** | 每个角色只拥有完成本职工作所需的最小权限 |
| 4 | **异步优先** | 微信场景天然异步，长任务通过SSE实时推送进度 |
| 5 | **可观测** | 每次调度、每个决策可追溯、可审计 |

## 二、各 Agent 详细定义

### 2.1 老k — 中枢调度 Agent

| 属性 | 值 |
|------|-----|
| 文件名 | `lao-k.md` |
| 模式 | `primary` |
| 角色定位 | 项目经理 + 调度中枢 + 微信对外接口 |
| 推荐模型 | 强推理模型（deepseek-r1 / glm-5.2） |

**能力边界：**

```
✅ 读取代码库（理解项目上下文）
✅ 调度所有子agent（通过Task工具）
✅ 拆解需求为多步骤任务
✅ 汇总子agent结果并二次加工
✅ 搜索网页/获取外部信息
✅ 使用todowrite追踪任务进度
✅ 提问用户澄清模糊需求

⚠️ 可写文件（仅用于生成调度报告/摘要，不写业务代码）
⚠️ 可执行shell（仅用于信息收集，如 git log、ls 等只读命令）

❌ 不写任何生产代码
❌ 不写测试代码
❌ 不做部署操作
❌ 不做UI设计
❌ 不直接做需求文档
```

**调度决策逻辑：**

```
收到微信消息
  │
  ├─ 纯问答类 → 老k直接回答（不调子agent）
  │
  ├─ 需求/功能类 → 产品PM → 根据需要串联后续角色
  │    └─ 需求文档 → UI设计 → 前端开发
  │                     └─ 后端开发 → 测试 → 运维
  │
  ├─ Bug修复类 → 测试(定位) → 对应开发agent
  │
  ├─ 部署/运维类 → 运维
  │
  └─ 模糊/复杂类 → 先问用户澄清 → 再调度
```

**通讯协议（Task工具调用格式）：**

```json
{
  "prompt": "<角色身份>. <具体任务描述>. <上下文>. <输出要求>.",
  "subagent_type": "product | ui | frontend | backend | tester | ops",
  "description": "<一句话任务摘要>"
}
```

### 2.2 产品PM Agent

| 属性 | 值 |
|------|-----|
| 文件名 | `product.md` |
| 模式 | `subagent` |
| 角色定位 | 产品经理 / 需求分析师 |
| 推荐模型 | 强推理 + 强语言表达（glm-5.2） |

**能力边界：**

```
✅ 分析需求并结构化
✅ 撰写PRD（产品需求文档）
✅ 拆解用户故事 / 验收标准
✅ 竞品调研（websearch/webfetch）
✅ 输出功能规格说明

⚠️ 可读写文件（仅文档类 .md/.txt）

❌ 不写任何代码（.js/.ts/.py/.go 等）
❌ 不执行shell命令
❌ 不做UI设计（交给UI Agent）
❌ 不写测试
❌ 不做部署
```

**输出物规范：**

```markdown
# 功能名称

## 背景与目标
## 用户故事
- 作为 [角色]，我想要 [功能]，以便 [价值]
## 功能描述
## 交互流程
## 验收标准
- [ ] 标准1
- [ ] 标准2
## 非功能需求（性能/安全等）
## 依赖项
```

**通讯协议：**

```
输入：老k下发的 Task(prompt)
  格式：「作为产品PM，请基于以下需求撰写PRD：<需求描述>。请按照标准PRD模板输出。」

输出：PRD文档（Markdown格式）
  直接作为Task返回值传回老k
```

### 2.3 UI设计 Agent

| 属性 | 值 |
|------|-----|

| 模式 | `subagent` |

```
✅ 设计Design Token（颜色/字体/间距）
✅ 输出响应式适配方案

⚠️ 可读写文件（仅 .md 设计文档）
⚠️ 可访问设计资源网站（webfetch）

❌ 不写CSS/样式代码（交给前端）
❌ 不写任何业务代码
❌ 不执行shell
❌ 不做需求定义（交给产品）
❌ 不做测试
❌ 不做部署
```

**输出物规范：**

```markdown
# [页面/组件] 设计方案

## 设计目标
## 页面结构（信息架构）
## 组件树
## 交互说明
- 状态流转图（文字描述）
- 用户操作 → 系统响应
## 视觉规范
- 颜色方案
- 字体方案
- 间距系统
## 响应式断点
## 可访问性考虑
```

**通讯协议：**

```
输入：Task(prompt)
  格式：「作为UI设计师，请基于以下PRD设计页面方案：<PRD内容>。如附带参考图，请分析并融入设计。」

输出：设计规范文档（Markdown）
  可附带对参考图的分析结果
```

## 四、通讯流程详解

### 4.1 标准新功能开发流程

```
用户(微信)："帮我做一个用户注册功能"
    │
    ▼
微信Bot → POST /session/:id/message (agent=lao-k)
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 老k 接收消息                                         │
│ 1. 分析意图 → 功能开发类                              │
│ 2. 拆解为5个阶段任务                                  │
│ 3. todowrite 创建任务清单                             │
└─────────────────────────────────────────────────────┘
    │
    ▼ 第1阶段：需求分析
┌─────────────────────────────────────────────────────┐
│ 老k → Task(subagent_type="product")                  │
│ prompt: "作为产品PM，请分析以下需求并撰写PRD：          │
│          用户注册功能，支持手机号+邮箱注册..."            │
│                                                      │
│ 产品PM → 返回: PRD文档                                 │
└─────────────────────────────────────────────────────┘
    │
    ▼ 第2阶段：UI设计
┌─────────────────────────────────────────────────────┐
│ 老k → Task(subagent_type="ui")                       │
│ prompt: "作为UI设计师，请基于以下PRD设计页面方案：<PRD>" │
│                                                      │
│ UI → 返回: 注册页面设计规范                             │
└─────────────────────────────────────────────────────┘
    │
    ▼ 第3阶段：前后端并行开发
┌───────────────────────┐  ┌──────────────────────┐
│ 老k → Task(backend)    │  │ 老k → Task(frontend)  │
│ "基于PRD实现注册API"    │  │ "基于PRD+UI设计实现    │
│                       │  │  注册页面前端"         │
│ 后端 → 返回: API代码    │  │ 前端 → 返回: 页面代码  │
└───────────────────────┘  └──────────────────────┘
    │                           │
    ▼ 第4阶段：测试                │ (等待前端完成)
┌─────────────────────────────────────────────────────┐
│ 老k → Task(subagent_type="tester")                   │
│ prompt: "测试注册功能，PRD：<验收标准>，                 │
│          后端代码：<文件路径>，前端代码：<文件路径>"       │
│                                                      │
│ 测试 → 返回: 测试报告                                  │
│  (如有失败 → 老k调对应开发Agent修复 → 回归测试)          │
└─────────────────────────────────────────────────────┘
    │
    ▼ 第5阶段：部署
┌─────────────────────────────────────────────────────┐
│ 老k → Task(subagent_type="ops")                      │
│ prompt: "注册功能已测试通过，请配置CI/CD并部署"          │
│                                                      │
│ 运维 → 返回: 部署报告                                  │
└─────────────────────────────────────────────────────┘
    │
    ▼ 老k汇总所有结果
```

### 5.1 技术选型

### 5.3 核心流程

```javascript
// 伪代码 - 消息处理主流程
async function handleWechatMessage(userId, message) {
  // 1. 获取或创建 OpenCode 会话
  let sessionId = await sessionManager.get(userId);
  if (!sessionId) {
    const res = await opencodeClient.post('/session', {
      title: `微信用户-${userId}`
    });
    sessionId = res.id;
    await sessionManager.set(userId, sessionId);
  }

  // 2. 发送即时确认
  await wechatClient.send(userId, "⏳ 收到，正在处理中...");

  // 3. 异步发送到 OpenCode（指定老k agent）
  await opencodeClient.post(`/session/${sessionId}/prompt_async`, {
    agent: "lao-k",
    parts: [{ type: "text", text: message }]
  });

  // 4. 监听SSE事件，实时推送进度
  const eventSource = new EventSource(
    `${OPENCODE_URL}/event`
  );
  eventSource.onmessage = async (event) => {
    const data = JSON.parse(event.data);
    if (data.type === "message.completed" && data.sessionID === sessionId) {
      const reply = extractReply(data);
      await wechatClient.send(userId, reply);
      eventSource.close();
    }
    if (data.type === "message.partial" && data.sessionID === sessionId) {
      // 可选：流式推送中间结果
    }
  };
}
```

## 七、关键设计决策与推演

### 7.1 为什么选择 Hub-Spoke 而非网状通讯？

**网状模型（Agent互相调用）的风险：**
```
前端 → 调用后端了解API → 后端又调前端确认UI规范 → 前端又调测试验证 →
测试发现Bug → 调后端修复 → 后端又调前端改接口 → 无限循环
```

**Hub-Spoke 模型的优势：**
- 老k作为唯一调度者，掌握全局状态
- 每次调度都有明确的输入和输出
- 死循环不可能发生（子agent无法调用其他agent）
- 调试简单：查看老k的调度日志即可还原整个流程

### 7.2 老k的调度决策如何保证准确？

老k在调度前会：
1. **读AGENTS.md**理解项目上下文
2. **用glob/grep搜索代码**理解当前状态
3. **分析消息意图**匹配到对应流程模板
4. **组装上下文**把PRD、设计稿、代码路径等作为prompt注入
5. **选择模型**（当前统一用glm-5.2，未来可按角色选不同模型）

### 7.3 子Agent之间的间接协作如何实现？

虽然子Agent不能直接通讯，但通过老k的中转实现间接协作：

```
老k → Task(后端) "实现API" → 后端返回: API文档（接口路径+参数+返回值）
老k → Task(前端) "实现页面，参考以下API文档：<后端返回的API文档>"
```

关键是**老k负责在调度链中传递中间产物**，每个Agent的输出成为下一个Agent的输入上下文。

### 7.4 失败处理策略

```
子Agent执行失败
    │
    ├─ 代码编译/构建失败
    │   → 老k将错误日志反馈给同一个Agent重试（最多2次）
    │   → 仍失败则换Agent处理或反馈给用户
    │
    ├─ 测试不通过
    │   → 老k将测试报告发给对应开发Agent修复
    │   → 修复后重新测试（最多3轮）
    │   → 仍不通过则反馈给用户人工介入
    │
    ├─ 部署失败
    │   → 老k将错误信息发给运维Agent排查
    │   → 尝试回滚
    │   → 反馈给用户
    │
    └─ Agent超时/无响应
        → 老k中止该任务
        → 反馈给用户并建议替代方案
```

### 7.5 安全考量

| 风险 | 防护措施 |
|------|---------|
| Agent执行危险shell命令 | bash权限按角色分级，危险操作需ask确认 |
| Agent访问不应修改的文件 | edit权限按目录/扩展名精细控制 |
| 无限循环调度 | 子agent的task权限为deny，不可调用其他agent |
| 微信消息泄露 | 会话隔离，每个用户独立session |
| 代码注入 | 所有用户输入经老k审查后再分发 |
